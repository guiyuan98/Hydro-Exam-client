# 面向 OI 竞赛的在线评测系统安全考试客户端设计与实现

## 摘要

随着程序设计教学和信息学竞赛训练规模的扩大，在线评测系统已经成为算法教学、作业管理和竞赛组织的重要基础设施。传统浏览器访问模式虽然部署方便，但在受控机房考试中难以限制切屏、复制粘贴、打开外部 IDE 或聊天软件等行为。针对这一问题，本文设计并实现了一套面向 OI 竞赛的安全考试客户端。系统采用 Hydro OJ 作为服务端，保留其题库、比赛、提交、排名和远程隐藏数据判题能力；客户端采用 Windows 桌面软件形式，主窗口直接加载自部署 Hydro 页面，同时自动打开本地 C++ 编辑器和样例运行器，实现本地样例测评与远程正式判题分离。客户端默认启用全屏置顶、复制粘贴拦截、剪贴板清空、失焦锁定、黑名单进程检测和教师密码解锁等监考能力，并预留 Hydro 插件化监考审计扩展方向。

关键词：在线评测；Hydro OJ；OI 竞赛；考试客户端；本地样例测评；监考审计

## 1 绪论

OI 竞赛要求选手在较长时间内阅读题目、编写程序、运行样例并多次提交。正式考试还要求隐藏测试数据不能泄露，学生不能随意切换到浏览器、聊天工具、网盘或外部 IDE。仅使用普通网页 OJ 难以约束这些行为，因此需要一个与 OJ 深度结合的桌面考试客户端。

本课题的目标是构建“Hydro OJ 服务端 + Windows 考试客户端 + 本地 C++ 编辑器”的一体化考试环境。Hydro 负责用户、题库、比赛、提交、判题和排名；客户端负责受控入口、本地样例运行和基础监考。

## 2 相关技术

Hydro OJ 是一套开源在线评测系统，支持题目管理、比赛组织、提交记录、排名统计和沙箱判题。Hydro 还提供插件扩展能力，适合作为监考审计模块的后续接入点。

Electron 基于 Chromium 和 Node.js，适合将 Web OJ 页面封装成桌面软件。本系统利用 Electron 主窗口直接加载 Hydro 页面，避免重复实现 OJ 前端；同时使用 Node.js 调用本地 g++ 编译器，实现样例编译运行。

## 3 需求分析

教师需要部署 OJ、创建超级管理员、录入题目、创建比赛、设置参赛用户并查看排名。学生需要在统一客户端内完成读题、编辑、运行样例和正式提交。监考需求包括禁止切屏、禁止复制粘贴、检测黑名单进程和锁定考试窗口。

该方案属于用户态客户端监考，不能声称绝对防作弊。论文中应限定适用场景：受控 Windows 机房、管理员权限、非内核级对抗条件。

## 4 系统设计

系统由三部分组成：Hydro OJ 服务端、Windows 考试客户端、本地 MinGW/g++ 编译环境。Hydro 服务端保存正式题目数据、隐藏测试数据和提交记录；客户端只负责加载 Hydro 页面和运行样例，不保存正式测试数据。

客户端启动后进入正式考试模式：主窗口全屏加载 `serverBaseUrl` 指向的 Hydro 页面，工具窗口自动打开 C++ 编辑器。学生可运行样例检查程序行为，正式提交仍在 Hydro 页面完成。

## 5 系统实现

服务端部署采用 WSL2 中的一键安装脚本，当前部署包含 `hydrooj@5.0.3`、`@hydrooj/hydrojudge@4.0.4`、MongoDB 和 Caddy。管理员账号通过 Hydro 注册流程创建，并使用 `hydrooj cli user setSuperAdmin 2` 提升为超级管理员。

客户端配置文件为 `client/oj-shell/gyoj-shell.json`，其中 `serverBaseUrl` 默认设置为 `http://192.168.1.149`。调试时可使用 `--oj-url=http://localhost` 临时覆盖地址。

监考实现包括：全屏置顶、kiosk 模式、禁用右键菜单、拦截复制粘贴和切屏快捷键、定时清空剪贴板、失焦锁定、黑名单进程检测和教师密码解锁。

## 6 测试

测试内容包括 Hydro 服务是否启动、本机是否能访问中文首页、超级管理员是否可登录、学生端是否默认打开 Hydro、编辑器是否自动打开、本地 A+B 样例是否能编译运行、切屏和复制粘贴是否触发限制、正式提交是否进入 Hydro 远程判题队列。

## 7 总结

本文设计并实现了一套基于 Hydro OJ 的 OI 考试客户端原型。系统保留成熟 OJ 的远程判题能力，将本地编辑器和样例运行器集成到桌面客户端，并实现了基础监考限制。后续可继续完善 Hydro 插件化监考看板、IP/机器绑定、远程解锁、自更新、客户端签名和网络白名单。
